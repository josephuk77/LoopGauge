# 벤치마크 방법론

[English](benchmark-methodology.md) | 한국어

LoopGauge는 품질 제약을 만족하는 범위에서 비용을 최적화합니다. 최종 holdout 결과를 확인하기 전에 작업 분포, grader, 비용과 종료 규칙을 정의해야 신뢰할 수 있는 벤치마크가 됩니다.

## 근거 수준

1. **단위 테스트**는 구현 정확성, 공급자 격리, 점수 계산과 비용 계산을 검증합니다.
2. **합성 재현**은 실제 모델 성능을 주장하지 않고 리포트 동작을 보여줍니다.
3. **공개 실사용 API 벤치마크**는 라이선스가 명확하고 재현 가능한 작업 세트에서 공급자 호출을 측정합니다.
4. **비공개 shadow pilot**은 후보 출력을 고객에게 전달하지 않은 상태로 실제 작업 흐름을 측정합니다.

모든 리포트에는 근거 수준을 눈에 띄게 표시해야 합니다.

## 데이터셋 분할

- 의도적으로 고른 쉬운 예시가 아니라 대표성 있는 작업을 사용합니다.
- 프롬프트·하네스 개발 작업과 최종 평가용 holdout 세트를 분리합니다.
- 제외 항목과 실패를 숨기지 말고 기록합니다.
- 모델 분산이 의사결정을 바꿀 수 있다면 여러 번 반복합니다.
- 데이터셋 라이선스와 비식별화 규칙을 공개합니다.

## 품질 정의

실행 전에 다음을 정의합니다.

- 필수 탈락 조건
- 결정론적 테스트와 grader
- 평가 기준별 가중치
- 주관적 출력에 대한 사람 검토 절차
- 기준 모델과 모델 snapshot
- 기준 품질 대비 최소 비율
- 일치도를 참고 지표로만 쓸지, 후보 통과 조건으로 쓸지 여부

teacher 모델 하나만 judge로 사용하지 마세요. 결정론적 검사, 정보를 가린 사람 검토 또는 독립적인 근거가 있는 grader를 함께 사용해야 합니다.

## 비용 정의

API 호출당 비용이 아니라 성공하거나 승인된 작업당 비용을 사용합니다.

```text
전체 비용 = 입력 + 출력 + 캐시 + 도구 + 재시도 + judge + 승격 + 실패 작업
성공당 비용 = 전체 정책 비용 / 성공한 작업 수
```

비즈니스 작업 흐름에서는 사람의 수정 시간을 별도로 보고하거나, 공개한 시간당 비용으로 환산합니다. 모든 리포트에 공급자 가격표 기준 시각을 기록합니다.

## 탐색과 종료

다음을 사전에 등록합니다.

- 최적화 예산
- 호출당·작업당 예산
- 최대 반복 횟수
- 무개선 종료 횟수
- 중단 동작
- 승격과 fallback 정책

공개 결과를 개선하려는 목적으로 선언한 한도를 초과해서는 안 됩니다.

## 공개 리포트 최소 요건

- 정확한 실행 명령과 LoopGauge 커밋
- 프로젝트·작업 버전
- 모델 ID와 가격표 기준 시각
- 표본 수와 반복 횟수
- 기준 품질과 비용
- 모든 후보의 성공률, 품질, 차이 점수와 총비용
- 최적화 비용과 이를 포함한 절감률
- 손익분기 작업 수
- 비식별화된 원시 JSON 또는 JSONL 근거
- 한계, 이해관계와 부정적 결과
